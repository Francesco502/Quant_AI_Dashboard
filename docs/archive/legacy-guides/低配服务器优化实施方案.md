> 版本：v1.0 | 适用项目版本：v3.0.0 | 编写日期：2026-02-24

## 一、背景与目标

### 1.1 当前服务器配置

| 项目 | 规格 |
|------|------|
| CPU | 2 核 |
| 内存 | 2 GB |
| 磁盘 | 云盘（具体容量视实际而定） |

### 1.2 当前部署资源消耗（标准模式）

标准部署使用 4 个 Docker 容器（backend / frontend / daemon / nginx），总内存需求约 **3～4 GB**，远超 2GB 上限。

| 资源消耗来源 | 估算内存占用 | 说明 |
|---|---|---|
| Uvicorn workers × 2 | 100–200 MB | 每个 worker 独立进程 |
| Next.js standalone 运行时 | 100–150 MB | Node.js 进程常驻 |
| Daemon 守护进程 | 50–100 MB | 独立 Python 进程 |
| ML 模型加载（单个） | 150–500 MB | LSTM 最高，LightGBM 最低 |
| SQLite 内存映射 + 页缓存 | ~288 MB | mmap_size=256MB + cache_size=32MB |
| 多级缓存 L1 (1000 项) | 50–100 MB | 内存 LRU 缓存 |
| LRU 缓存 | ≤50 MB | 限制 50MB 上限 |
| Nginx | ~10 MB | Alpine 镜像，开销极小 |
| **合计** | **~3–4 GB** | **超出 2GB 上限** |

### 1.3 优化目标

- 将总内存占用降至 **800 MB – 1.2 GB**，预留系统开销
- 保持所有业务功能可用（数据获取、AI 预测、模拟交易、回测、风控）
- 保证服务稳定性，不因资源不足 OOM 崩溃

## 三、方案详细说明

### 3.1 P0 — 前端静态导出（节省 ~150 MB）

#### 现状

```typescript
// web/next.config.ts
const nextConfig: NextConfig = {
  output: "standalone",  // 需要 Node.js 运行时常驻
};
```

前端以 `standalone` 模式部署，需要一个完整的 Node.js 进程常驻内存（100–150 MB），并单独占用一个 Docker 容器。

#### 优化方案

| 步骤 | 操作 | 说明 |
|------|------|------|
| 1 | 修改 `web/next.config.ts`，将 `output` 改为 `"export"` | 构建时生成纯静态 HTML/CSS/JS |
| 2 | 调整构建命令 `next build` | 输出到 `web/out/` 目录 |
| 3 | Nginx 直接托管 `out/` 静态文件 | 去掉 frontend 容器 |
| 4 | API 请求通过 Nginx `proxy_pass` 反向代理到 FastAPI | 保持 `/api` 路由不变 |

#### 前置检查

需确认以下几点：

- [ ] 前端不依赖 Next.js Server Actions 或 API Routes（当前 API 全部走 FastAPI，确认可行）
- [ ] 前端不依赖 SSR/ISR（当前主要为 CSR 客户端渲染，确认可行）
- [ ] 图片优化（`next/image`）需改为 `unoptimized: true` 或使用外部 loader

#### Nginx 配置示例

```nginx
server {
    listen 80;

    # 静态文件
    location / {
        root /app/web/out;
        index index.html;
        try_files $uri $uri.html $uri/ /index.html;
    }

    # API 反向代理
    location /api/ {
        proxy_pass http://127.0.0.1:8685;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
    }

    # WebSocket
    location /ws/ {
        proxy_pass http://127.0.0.1:8685;
        proxy_http_version 1.1;
        proxy_set_header Upgrade $http_upgrade;
        proxy_set_header Connection "upgrade";
    }
}
```

#### 风险与回退

- **风险**：`next/image` 组件在 `export` 模式下需要配置 `unoptimized`
- **回退**：如需恢复，将 `output` 改回 `"standalone"` 并重建 frontend 容器

### 3.2 P0 — ML 模型资源严格管控（节省 ~300 MB）

#### 现状

```python
# core/ml_model_manager.py
class MLModelManager:
    def __init__(
        self,
        max_memory_mb: float = 500.0,       # 最大内存 500MB
        auto_unload_interval: float = 60.0,  # 每分钟检查
    ):
```

```python
# core/ml_model_manager.py — ModelInfo
auto_unload_after: Optional[float] = 300.0  # 5分钟后自动卸载
```

单个 LSTM/GRU 模型即可占用 200–500 MB 内存，在 2GB 环境下加载一个 LSTM 就可能导致 OOM。

#### 优化方案

| 参数 | 当前值 | 优化值 | 说明 |
|------|--------|--------|------|
| `max_memory_mb` | 500 MB | **200 MB** | 全局模型内存上限 |
| `auto_unload_after` | 300 秒（5 分钟） | **120 秒（2 分钟）** | 更快释放不活跃模型 |
| `auto_unload_interval` | 60 秒 | **30 秒** | 更频繁检查过期模型 |

#### 禁用重型模型

在 2GB 环境下，**禁用 LSTM/GRU 深度学习模型**，仅保留轻量模型：

| 模型 | 单模型内存 | 2GB 环境 | 说明 |
|------|-----------|---------|------|
| LightGBM | 30–80 MB | ✅ 保留 | 内存最低，速度快 |
| XGBoost | 50–200 MB | ✅ 保留 | 精度高，内存适中 |
| Prophet | 50–150 MB | ✅ 保留 | 时序预测标配 |
| Random Forest | 50–100 MB | ✅ 保留 | sklearn 原生支持 |
| LSTM/GRU | 200–500 MB | ❌ 禁用 | 内存过高，PyTorch 运行时本身也占用大量内存 |
| ARIMA/SARIMA | 10–30 MB | ✅ 保留 | statsmodels，极轻量 |

#### 具体修改项

1. **`core/ml_model_manager.py`**：调降 `max_memory_mb` 和 `auto_unload_after`
2. **`daemon_config.json`**：将 `training.model_type` 默认值从可选 LSTM 改为 `xgboost` 或 `lightgbm`
3. **`core/advanced_forecasting.py`**：添加环境变量开关 `DISABLE_HEAVY_MODELS=true`，当开启时跳过 LSTM/GRU 相关调用，返回 fallback 到 XGBoost
4. **`requirements.txt`**：将 `torch>=2.1.0` 标记为可选（注释掉），在低配环境下不安装 PyTorch

#### 移除 PyTorch 的额外收益

PyTorch 本身安装包约 700MB+，仅 `import torch` 就会占用 ~100MB 内存。在 2GB 环境下移除 PyTorch 是性价比最高的单项优化。

```python
# requirements.txt 修改
# ========== 深度学习（低配环境可注释） ==========
# torch>=2.1.0              # PyTorch LSTM/Transformer（需 4GB+ 内存）
```

### 3.3 P0 — SQLite 内存参数调降（节省 ~250 MB）

#### 现状

```python
# core/database_pool.py — _create_connection()
conn.execute("PRAGMA cache_size=-32768")       # 32MB 页缓存
conn.execute("PRAGMA mmap_size=268435456")     # 256MB 内存映射
conn.execute("PRAGMA temp_store=MEMORY")       # 临时表存内存
```

`mmap_size=256MB` + `cache_size=32MB` 合计 288MB，在 2GB 环境下占比过高。

#### 优化方案

| PRAGMA 参数 | 当前值 | 优化值 | 节省内存 |
|---|---|---|---|
| `cache_size` | `-32768`（32 MB） | **`-4096`（4 MB）** | ~28 MB |
| `mmap_size` | `268435456`（256 MB） | **`33554432`（32 MB）** | ~224 MB |
| `temp_store` | `MEMORY` | **`FILE`** | 视临时表大小 |

#### 修改位置

文件：`core/database_pool.py` — `_create_connection()` 方法

```python
# 优化前
conn.execute("PRAGMA cache_size=-32768")       # 32MB
conn.execute("PRAGMA mmap_size=268435456")     # 256MB
conn.execute("PRAGMA temp_store=MEMORY")

# 优化后
conn.execute("PRAGMA cache_size=-4096")        # 4MB（足够日常查询）
conn.execute("PRAGMA mmap_size=33554432")      # 32MB（足够常用数据）
conn.execute("PRAGMA temp_store=FILE")         # 临时表写磁盘，节省内存
```

#### 性能影响评估

- `cache_size` 从 32MB 降到 4MB：对于本项目的查询模式（按 ticker+date 查询），4MB 足够覆盖热数据，性能影响极小
- `mmap_size` 从 256MB 降到 32MB：仅影响大范围顺序扫描场景，日常 CRUD 无感知
- `temp_store=FILE`：仅在包含 `GROUP BY`/`ORDER BY` 的复杂查询时有微小延迟

### 3.4 P1 — 后端服务精简（节省 ~100 MB）

#### 3.4.1 单 Worker 模式

```yaml
# docker-compose.yml — backend.command
# 优化前
command: python -m uvicorn api.main:app --host 0.0.0.0 --port 8685 --workers 2

# 优化后
command: python -m uvicorn api.main:app --host 0.0.0.0 --port 8685 --workers 1 --limit-concurrency 8
```

- 2 个 worker → 1 个 worker：节省约 50–100 MB（一个完整 Python 进程的开销）
- `--limit-concurrency 8`：限制最大并发请求数，防止突发流量压垮服务

#### 3.4.2 数据获取并发降级

当前 `core/data_service.py` 中多处使用 `ThreadPoolExecutor`，最高 8 并发。

| 位置 | 当前 `max_workers` | 优化值 | 说明 |
|------|-------------------|--------|------|
| `data_service.py:621` — 批量价格获取 | `min(len, 8)` | **`min(len, 3)`** | 主要并发入口 |
| `data_service.py:758` — AlphaVantage | `min(len, 4)` | **`min(len, 2)`** | 有 API 限速 |
| `data_service.py:874` — Binance | `min(len, 4)` | **`min(len, 2)`** | 减少内存压力 |
| `data_service.py:903` — Tushare | `min(len, 4)` | **`min(len, 2)`** | 减少内存压力 |
| `data_service.py:1046` — 其他 | `min(len, 4)` | **`min(len, 2)`** | 减少内存压力 |
| `market_scanner.py:92` — 扫描器 | `4` | **`2`** | 减少内存压力 |

#### 3.4.3 合并 Daemon 进程到主进程（可选）

将 Daemon 独立进程合并到 FastAPI 主进程中，使用 `asyncio` 定时任务替代独立进程：

```python
# 方案：在 FastAPI lifespan 中启动后台调度器
from contextlib import asynccontextmanager
import asyncio

@asynccontextmanager
async def lifespan(app: FastAPI):
    task = asyncio.create_task(run_scheduler())
    yield
    task.cancel()
```

**优点**：节省一个独立 Python 进程（50–100 MB）
**缺点**：主进程崩溃时定时任务也中断
**建议**：如内存极度紧张再实施，否则保留独立 Daemon 更稳健

### 3.5 P1 — 缓存层瘦身（节省 ~50 MB）

#### 3.5.1 多级缓存 L1 缩容

```python
# core/multi_level_cache.py
# 优化前
self.l1_cache = MemoryCache(max_size=1000)

# 优化后
self.l1_cache = MemoryCache(max_size=200)
```

从 1000 项缩减至 200 项，保留最热数据，其余走 L2 磁盘缓存。

#### 3.5.2 LRU 缓存缩容

```python
# core/lru_cache.py — LRUCache.__init__
# 优化前
max_size: int = 100
max_memory_mb: Optional[float] = None  # 使用时可传入 50MB

# 优化后
max_size: int = 50
max_memory_mb: Optional[float] = 20.0  # 硬限 20MB
```

#### 影响评估

缓存缩容会导致缓存命中率下降，但由于 L2 磁盘缓存兜底，对用户感知的响应时间影响约为 +50–100ms（磁盘 I/O），可接受。

### 3.6 P1 — Daemon 定时任务优化（避免峰值 OOM）

#### 3.6.1 训练任务错峰

```python
# daemon_config.json
# 优化前
"training": {
    "time": "16:00",       # 收盘后立即训练，与用户查看行情高峰重叠
}

# 优化后
"training": {
    "time": "02:00",       # 凌晨低峰执行
}
```

#### 3.6.2 数据更新减频

```python
# daemon_config.json
# 优化前
"data_update": {
    "interval_minutes": 60,     # 每小时更新
}

# 优化后
"data_update": {
    "interval_minutes": 120,    # 每 2 小时更新
}
```

#### 3.6.3 训练限流 — 串行化

在 `core/training_pipeline.py` 中确保：

- 每次只训练 **1 个标的** 的 **1 个模型**（串行）
- 训练完毕后主动 `gc.collect()` 释放内存
- 训练前检查系统可用内存，低于阈值时跳过

```python
import psutil, gc

def safe_train(ticker, model_type):
    mem = psutil.virtual_memory()
    if mem.available < 400 * 1024 * 1024:  # 可用内存 < 400MB 时跳过
        logger.warning(f"内存不足({mem.available // 1024 // 1024}MB)，跳过 {ticker} 训练")
        return None
    
    result = train_single_model(ticker, model_type)
    
    gc.collect()  # 强制回收
    return result
```

#### 3.6.4 训练任务内存熔断

在 Daemon 主循环中加入内存水位监控：

```python
def memory_guard(threshold_percent=80):
    """内存使用超过阈值时暂停任务"""
    mem = psutil.virtual_memory()
    if mem.percent > threshold_percent:
        logger.warning(f"内存使用率 {mem.percent}%，超过阈值 {threshold_percent}%，暂停任务")
        gc.collect()
        time.sleep(30)
        return False
    return True
```

### 3.7 P2 — Docker 部署优化（节省 ~100 MB）

#### 3.7.1 单容器部署

将 4 个容器合并为 **1 个容器**，用 Supervisor 管理内部进程，消除容器运行时的重复开销。

```yaml
# docker-compose.yml（优化版）
services:
  quant-app:
    build:
      context: .
      dockerfile: Dockerfile.optimized
    container_name: quant-app
    ports:
      - "8686:80"
    volumes:
      - ./data:/app/data
      - ./logs:/app/logs
      - ./strategies:/app/strategies
      - ./models:/app/models
    environment:
      - PYTHONUNBUFFERED=1
      - DISABLE_HEAVY_MODELS=true
    deploy:
      resources:
        limits:
          memory: 1800m
          cpus: "2.0"
    restart: unless-stopped
```

#### 3.7.2 内存限制与保护

| 配置 | 值 | 说明 |
|------|-----|------|
| `memory` | `1800m` | 预留 200MB 给宿主机系统 |
| `cpus` | `"2.0"` | 不超过 2 核 |
| `oom-kill-disable` | `false`（默认） | 允许 OOM Killer 生效，避免整机卡死 |

#### 3.7.3 容器内进程结构

```
Supervisor（PID 1）
├── Nginx（静态文件 + 反向代理）
├── Uvicorn（FastAPI，单 Worker）
└── Daemon（定时任务，可选）
```

## 九、附录：可选进阶优化

以下优化需要更多代码改动，可在基础方案稳定运行后考虑：

| 优化项 | 说明 | 收益 |
|--------|------|------|
| 使用 ONNX Runtime 替代 sklearn | 模型推理速度提升 2–5x，内存更低 | 中 |
| 引入 Redis 替代内存缓存 | 缓存独立于进程，重启不丢失 | 中（但需额外服务） |
| 数据分页加载 | 大批量数据按需加载，避免一次性读入 | 低-中 |
| API 响应缓存（HTTP Cache-Control） | 减少重复计算 | 低 |
| 使用 `uvloop` 替代默认事件循环 | 异步性能提升 2–4x | 低 |
| 数据库迁移到 PostgreSQL | 更好的并发支持（需额外内存） | 不建议（2GB 下反而更耗资源） |
