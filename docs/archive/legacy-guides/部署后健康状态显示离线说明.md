# 部署后「系统健康」显示离线说明

## 现象

在「系统设置 → 服务状态」中看到：

- **API 服务 (API Server)**：离线 (Offline)
- **数据馈送 (Data Feeder)**：未连接
- **信号引擎 (Signal Engine)**：空闲 (Idle)

## 原因说明

### 1. API 服务 / 数据馈送 显示离线或未连接

前端在**浏览器里**请求健康接口：`GET ${API_BASE_URL}/health`。

- `API_BASE_URL` 来自构建时的环境变量 **`NEXT_PUBLIC_API_URL`**。
- 若构建时未设置或设为 `http://127.0.0.1:8685/api`，则前端会请求 **用户本机** 的 `127.0.0.1:8685`，而不是你部署的服务器。
- 用户通过 `http://服务器IP:8686` 打开页面时，浏览器会去访问「用户自己电脑」上的 8685 端口，自然失败，因此显示 **API 离线**；数据馈送依赖 API，也会显示 **未连接**。

### 2. 信号引擎显示「空闲」

当前「信号引擎」为固定文案，未对接后端定时任务状态，因此始终显示「空闲 (Idle)」。v2.0.1 会优化说明或对接真实状态。

## 正确部署方式（修复「离线」）

### 单容器部署（Nginx 反代 API，推荐）

前端和 API 同域访问（例如 `http://服务器:8686` 下，API 为 `/api`）。构建镜像时**必须**让前端使用相对路径 `/api`：

- 在 **Docker 构建** 时设置：`NEXT_PUBLIC_API_URL=/api`
- 仓库内 `Dockerfile.optimized` 已包含：`ENV NEXT_PUBLIC_API_URL=/api`，用该 Dockerfile 构建出的镜像即可。

若你是**本地构建**再推送到服务器，请确保构建时带上：

```bash
export NEXT_PUBLIC_API_URL=/api
cd web && npm run build
```

或使用项目根目录的 `docker compose build`（会按 Dockerfile.optimized 用 `/api` 构建前端）。

### 前后端分离部署（API 单独域名或端口）

若前端和 API 不在同域（例如前端 `https://dashboard.example.com`，API `https://api.example.com`），则构建前端时显式指定 API 的完整地址：

```bash
export NEXT_PUBLIC_API_URL=https://api.example.com/api
cd web && npm run build
```

### 如何确认当前前端用的 API 地址

在浏览器打开部署后的页面，按 F12 → Console，输入：

```javascript
// 若为 Next 构建，一般可在 __NEXT_DATA__ 或运行时请求中看到
console.log('API base:', process.env.NEXT_PUBLIC_API_URL || 'http://127.0.0.1:8685/api')
```

若输出为 `http://127.0.0.1:8685/api`，说明当前前端是用「本机 8685」构建的，在服务器或其它机器访问就会显示 API 离线，需要按上面方式用 `/api` 或正确域名重新构建并部署。

## 修复后自检

1. 用浏览器访问：`http://你的服务器:8686/api/health`，应返回 JSON（含 `status`、`version` 等）。
2. 再打开「系统设置 → 服务状态」并点击「刷新状态」，API 应显示为 **运行中**，数据馈送为 **已连接**。

## v2.0.1 优化计划

- 健康检查：当检测到当前访问域名不是 localhost 且配置为 127.0.0.1 时，自动用「当前站点同源 /api」重试，减少错误配置导致的「离线」。
- 健康检查：增加重试与超时，失败时在页面上给出更明确的提示（例如「请确认 NEXT_PUBLIC_API_URL 与部署方式一致」）。
- 信号引擎：补充说明文案或对接监控/定时任务状态接口，使状态更易理解。
