# v2.0.0 Release Hardening Design

**Scope:** Promote the current `v2.0.0-alpha.1` codebase to a stable `v2.0.0` release without expanding feature scope.

**Status:** Approved design baseline for direct-freeze release path.

## Goal

Use the existing `alpha.1` branch state as the release baseline and harden it into a formal `v2.0.0` release by freezing scope, closing security and compatibility gaps, and producing reproducible release evidence.

## Non-Goals

- No major new product features.
- No broad refactor unrelated to release stability.
- No further expansion of experimental modules just because they already exist in the tree.

## Release Strategy

The release will follow a direct-freeze path:

1. Freeze feature scope at the current codebase.
2. Fix only release-critical issues.
3. Re-verify the full release matrix.
4. Publish `v2.0.0` only after formal release criteria are met.

This avoids extending the risk surface while still allowing targeted hardening inside the already changed areas.

## Required Release Criteria

The codebase must satisfy all of the following before `v2.0.0` is tagged:

1. Security defaults are safe for a public release.
2. Version metadata and release-facing docs consistently describe `v2.0.0`.
3. API/configuration behavior is documented and stable enough for public use.
4. Docker, backend, frontend, and key end-to-end flows pass verification.
5. Release notes, upgrade notes, and rollback guidance are present.

## Workstreams

### 1. Scope Freeze

Review the current release candidate and classify new additions into:

- stable in `v2.0.0`
- experimental but shipped
- excluded from `v2.0.0` guarantees

This classification should be reflected in release docs so the formal version does not over-promise support.

### 2. Security Hardening

The current tree includes at least two formal-release blockers:

- `env.example` contains a real-looking API key value instead of a placeholder
- the auth layer auto-creates a default admin account with a weak password

For `v2.0.0`, secrets examples must be placeholders only, and first-run authentication must not silently create a known default credential in production-like environments.

### 3. Compatibility and Configuration Review

Validate and document:

- required vs optional environment variables
- production-safe defaults
- public API groups that are intended to remain stable
- fallback behavior when Tushare/AkShare/LLM services are unavailable

### 4. Verification and Release Evidence

Re-run the release matrix after hardening and record:

- Python compile pass
- unit/integration/E2E test results
- frontend lint/build
- Docker build/up/health
- critical smoke flows

## Error Handling Expectations

Release-critical flows must fail in a predictable and supportable way:

- invalid configuration should produce explicit error messages
- unavailable optional providers should degrade gracefully
- auth failures should be explicit and not leak secrets
- release docs should explain expected degraded behavior where applicable

## Testing Requirements

The hardening work must include tests for any changed release-critical behavior, especially:

- secure auth bootstrap behavior
- safe configuration defaults
- version metadata and release smoke expectations

## Documentation Requirements

The release line must include:

- final release notes for `v2.0.0`
- upgrade notes from `alpha.1`
- rollback guidance
- a release validation summary

## Immediate First Batch

The first implementation batch should focus on the highest-risk formal-release blockers:

1. remove real-looking secret values from examples and docs
2. replace default-admin bootstrap with a safer first-run strategy
3. define the formal-release document set and version line update path
