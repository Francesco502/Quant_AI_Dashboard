# Quant-AI Dashboard 项目优化方案

> 基于用户需求目标与现状分析，制定详细的项目升级优化方案

## 一、需求目标回顾

| 编号 | 需求描述 | 优先级 |
|------|----------|--------|
| 1 | 个人指定资产分析（股票、基金、黄金等），分析风险、涨跌 | 高 |
| 2 | 全市场找股（A股、港股），按策略筛选优质股票 | 高 |
| 3 | 模拟账户量化交易，验证策略，观察盈亏 | 高 |
| 4 | 回测功能 | 中 |
| 5 | 多用户登录，各用户独立配置 | 中 |
| 6 | 日线级别长期分析（不需要分钟线） | 高 |
| 7 | 服务器性能适配（2核CPU, 2GB RAM） | 高 |
| 8 | 参考daily_stock_analysis和dexter优秀设计 | 中 |

### 2.1 需求1：个人资产分析

#### 现有能力
| 模块 | 文件 | 功能 | 状态 |
|------|------|------|------|
| 多资产识别 | `core/data_service.py:identify_asset_type()` | 识别股票/基金/黄金/加密货币 | ✅ 已实现 |

| 技术信号 | `core/strategy_engine.py` | MA/RSI/MACD综合信号 | ✅ 已实现 |

#### 存在问题
| 问题 | 详情 | 影响 |
|------|------|------|
| **黄金数据源缺失** | 仅有yfinance支持，国内黄金（如AU99.99）数据源未集成 | 无法分析国内黄金 |

#### 改进建议

   - 检查所有数据表，确保有user_id字段
   - API层强制按user_id过滤数据

2. **新增用户配置相关表**
   ```sql
   -- 用户自选股
   CREATE TABLE user_watchlist (
       id INTEGER PRIMARY KEY,
       user_id INTEGER NOT NULL,
       ticker TEXT NOT NULL,
       added_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
       UNIQUE(user_id, ticker)
   );

   -- 用户策略配置
   CREATE TABLE user_strategy_config (
       id INTEGER PRIMARY KEY,
       user_id INTEGER NOT NULL,
       strategy_name TEXT NOT NULL,
       config_json TEXT NOT NULL,
       created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
   );

   -- 用户偏好设置
   CREATE TABLE user_preferences (
       id INTEGER PRIMARY KEY,
       user_id INTEGER UNIQUE NOT NULL,
       default_strategy TEXT,
       risk_tolerance TEXT DEFAULT 'moderate',
       notification_enabled INTEGER DEFAULT 0,
       preferences_json TEXT
   );
   ```

3. **创建用户配置API**
   - 创建 `api/routers/user_config.py`
   - 提供自选股、策略配置、偏好设置的CRUD接口

### 2.7 需求7：服务器性能适配（2核2GB）

#### 现有优化
| 优化项 | 文件 | 详情 |
|--------|------|------|
| 禁用重型模型 | `requirements.txt` | LSTM/GRU默认注释 |
| Docker资源限制 | `docker-compose.yml` | CPU 2核, 内存1.8GB |
| 环境变量 | `DISABLE_HEAVY_MODELS=true` | 运行时禁用 |

#### 存在问题
| 问题 | 详情 | 影响 |
|------|------|------|
| **内存占用偏高** | 多模块同时加载可能超限 | 服务崩溃风险 |
| **缺少内存监控告警** | 无主动内存预警机制 | 故障发现滞后 |
| **部分模块可精简** | 存在未使用或冗余代码 | 资源浪费 |
| **缓存策略不完善** | 数据缓存可能导致内存增长 | 内存泄漏风险 |

#### 改进建议
1. **内存监控与告警**
   - 在 `api/main.py` 健康检查中增加内存告警
   - 内存使用率>80%时记录警告日志
   - 考虑添加企业微信/飞书告警通知

2. **精简模块加载**
   - 使用懒加载（importlib.util）延迟加载重型模块
   - 移除或注释不需要的深度学习相关代码

3. **优化缓存策略**
   - 使用LRU缓存并设置maxsize
   - 定期清理过期缓存

## 六、风险与注意事项

### 6.1 技术风险

| 风险 | 影响 | 缓解措施 |
|------|------|----------|
| 内存溢出 | 服务崩溃 | 实现内存监控+自动重启 |
| 数据源失效 | 功能不可用 | 多数据源fallback机制 |
| LLM API限流 | AI分析失败 | 降级为技术分析，提示用户 |

### 6.2 数据风险

| 风险 | 影响 | 缓解措施 |
|------|------|----------|
| 数据延迟 | 分析不准确 | 提示数据更新时间 |
